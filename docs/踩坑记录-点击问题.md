## 问题描述

`Input.synthesizeTapGesture` 在 macOS 上正常运行，但在 Windows 上点击无响应。把 API 换成 `Input.dispatchMouseEvent` 后虽然解决了 Windows 点击问题，但是又被风控识别，无限循环验证。

## 核心原因：isTrusted 属性

这是两个 API 最关键的区别：

| API | isTrusted | 结果 |
|-----|-----------|------|
| `dispatchMouseEvent` | **false** | 被风控识别 |
| `synthesizeTapGesture` | **true** | 被识别为真实用户操作 |

### dispatchMouseEvent 为什么会被检测？

1. **isTrusted = false** — 表明事件由脚本触发，被风控直接拦截
2. **坐标异常** — 错误地设置 `PageX == ScreenX`，真实事件中这些坐标应该不同
3. **缺少移动轨迹** — 没有 `mousemove` 事件，鼠标瞬移到目标位置

### synthesizeTapGesture 为什么能绕过？

1. **isTrusted = true** — 在浏览器内核层面合成系统级手势事件
2. **完整事件链** — 自动生成 touchstart → touchend 等底层事件序列
3. **真实时序** — 模拟真实的时间间隔和事件顺序

## Windows 兼容解决方案

`synthesizeTapGesture` 默认使用 `gestureSourceType: 'default'`，在 Windows 上会被识别为触摸事件而无效。

**修复方法**：显式指定为鼠标事件

```javascript
Input.synthesizeTapGesture({
  x: 100,
  y: 100,
  gestureSourceType: 'mouse'  // 关键：指定为鼠标事件
})
```

## 总结

两者都是 CDP（Chrome DevTools Protocol）方法，但 `synthesizeTapGesture` 通过生成 `isTrusted=true` 的真实系统事件，成功绕过了 CF 对脚本模拟点击的检测。配合 `gestureSourceType: 'mouse'` 可同时兼容 macOS 和 Windows。
